Capability Summary

The reviewer should be able to understand what this capability does before seeing any architectural detail. The business language stays in front; technical context is kept short and concrete.

Start Here

Business outcome

Convert a checked shopping cart into a committed order, reserve inventory, take payment authorization, and queue customer confirmation without forcing the reviewer to inspect the legacy code.

Primary actors

  • B2B customer placing an order through the storefront
  • Call-center agent submitting an assisted order
  • Pricing and inventory services used as supporting systems

Systems touched

  • Checkout UI
  • Order processor
  • Pricing rules engine
  • Inventory adapter
  • Payment gateway
  • Notification queue

Top review risks

  • Discount logic mixes customer contract pricing with promotional thresholds.
  • Gateway timeout handling may create duplicate or half-finished orders.
  • Manual price overrides may bypass audit requirements in some call-center flows.

This summary is intentionally short. It tells the reviewer what the capability is, what matters, and where to focus.

Business Requirements To Review

This is the main review surface. Requirement cards are short, decision-shaped, and evidence-backed. Reviewers should be able to confirm, correct, or defer each item without reading prose walls.

Primary Review Surface
REQ-OS-101 Needs review

Apply contract pricing first, then apply a 15% bulk discount only when order subtotal exceeds $500 and no manual override exists.

Critical Confidence: Medium Workflow: Pricing
Trigger

Customer or call-center agent submits the order after cart review.

Expected outcome

Pricing is recalculated once, using contract price where present and bulk discount only when threshold conditions are met.

Exception / concern

Manual override may bypass the threshold rule. Reviewer should confirm whether that is intended.

Why it was inferred

The rule appears across pricing helper code, a SQL promotion lookup, and a call-center override branch.

Source references
  • checkout/submitOrder.cfm
  • pricing/applyDiscount.cfc
  • sql/getContractPrice.sql
Why confidence is medium

Threshold logic is clear, but override handling branches differently in storefront and call-center flows.

REQ-OS-102 Confirmed

Reject order submission when the authenticated session has expired, and redirect the user to re-authenticate before any payment attempt.

Standard Confidence: High Workflow: Authentication
Trigger

User submits order with an invalid or missing checkout session.

Expected outcome

No order record or payment attempt occurs. User is routed back to login and returned to checkout after success.

Exception / concern

None in the current evidence set. This rule is stable and already low risk.

Why it was inferred

Guard clause occurs before order creation and before payment gateway invocation.

Source references
  • checkout/submitOrder.cfm
  • auth/sessionGuard.cfc
Why confidence is high

Behavior is consistent across both storefront and assisted ordering entry points.

REQ-OS-103 Needs review

If payment authorization times out after the gateway call is issued, the system should create a recoverable pending-payment order rather than silently failing.

Critical Confidence: Low Workflow: Payment
Trigger

Gateway request returns timeout or no callback within the allowed window.

Expected outcome

Order is recorded in a recoverable state so operations can reconcile or resume without duplicate charging.

Exception / concern

The legacy code shows both a draft-save path and a hard-fail path. This needs explicit reviewer confirmation.

Why it was inferred

A timeout branch writes to a recovery table in one path, but an older branch returns an error without persistence.

Source references
  • checkout/submitOrder.cfm
  • payment/gatewayAdapter.cfc
  • ops/saveFailedPayment.cfc
Why confidence is low

Behavior differs by gateway adapter version and may have drifted through partial hotfixes.

REQ-OS-104 Corrected

Reserve inventory only after payment authorization succeeds; if payment fails, release any temporary hold before returning control to the user.

Important Confidence: Medium Workflow: Inventory
Trigger

Payment response returns success, failure, or timeout after initial stock validation.

Expected outcome

Inventory reservation becomes durable only after payment success. Temporary hold is dropped on failure.

Exception / concern

Earlier draft said reservation happened before payment. Reviewer corrected that to a temporary hold model.

Why it was inferred

Inventory adapter performs a temporary hold during validation, then a final reserve call after gateway success.

Source references
  • inventory/checkAndHold.cfc
  • inventory/finalizeReservation.cfc
Correction history

SME review changed the wording from "reserve inventory before payment" to "temporary hold before payment."

REQ-OS-105 Deferred

Every manual price override by a call-center agent must generate an audit entry containing supervisor approval and reason code.

Important Confidence: Medium Workflow: Audit
Trigger

Call-center user manually edits price or applies an override code during order submission.

Expected outcome

System writes user, reason code, supervisor approval, timestamp, and order reference to audit storage.

Exception / concern

Evidence for supervisor approval is inconsistent. This is deferred pending SME confirmation.

Why it was inferred

Audit write is obvious, but supervisor approval appears only in some branches and in a help-text file.

Source references
  • callcenter/overridePrice.cfm
  • audit/writeAuditEvent.cfc
  • help/callcenter_override.txt
Why it is deferred

This rule matters for governance, but the current evidence set does not prove whether supervisor approval is mandatory.

REQ-OS-106 Needs review

Queue the confirmation email asynchronously after the order is committed so that email delivery failure does not block order creation.

Standard Confidence: High Workflow: Notification
Trigger

Order commit succeeds and order number is available.

Expected outcome

Email is queued for downstream processing. Order response returns even if the mail service is unavailable.

Exception / concern

Reviewer should confirm whether backorder status changes the template or suppresses immediate email.

Why it was inferred

Queue write occurs after order commit and before response serialization, with failure logged non-blockingly.

Source references
  • notifications/enqueueConfirmation.cfc
  • checkout/submitOrder.cfm
Why confidence is high

Behavior is consistently implemented and supported by the integration queue schema.

Open Questions And Risks

These are the items that should consume scarce SME time. Everything else should be reviewable from the pack without escalation.

Escalate Narrowly

Does timeout create a recoverable pending-payment order?

Critical

Current evidence shows both a recovery path and a hard-fail path. Reviewer decision is needed before target design can treat timeout recovery as required behavior.

Is supervisor approval mandatory for all manual overrides or only above a value threshold?

Important

The audit write is clear. The approval requirement is not. This should go only to the call-center SME, not the full reviewer group.

Should confirmation email be delayed when inventory is partially backordered?

Standard

Base queueing behavior is clear. Only the backorder variant needs confirmation, and it does not block the rest of the capability review.
